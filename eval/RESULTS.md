# 评测结果日志(RESULTS)

> 每一轮"评测 → 发现 → 优化 → 复测 → 保留版本"都在此追加记录,最新在最上。
> 目标:三维(长期记忆 / 情感支持 / 个性化与主动性)都没有明显短板,并在长期记忆上做出长板。
> 版本保留:每次优化生效后打 git tag(`eval/<维度>-cN`),便于回溯与对比。

指标说明:`Hit@k` = 前 k 条命中任一 gold 的比例;`MRR` = 首个 gold 命中名次的倒数均值。

## 第 3 轮 · 接公开基准 LoCoMo(现实检验)— 2026-06-17

**动机**:第 1/2 轮都在**自建集**上量,自说自话。接公开基准 **LoCoMo**(Snap Research,arXiv:2402.17753)做现实检验。
**口径**:**检索召回**——把对话每个 turn 作为一张卡灌入 LongTermMemory,对每题跑 `ltm.search`,看证据 turn(`evidence` 的 dia_id)是否进 top-k。category 5(对抗/不可答)排除。脚本 `eval/locomo_eval.py`,跑前 2 个样本(231 题,英文对话,c2 检索引擎)。

### 结果(top-k 敏感性)

| 类别 | n | H@5 | H@10 | H@20 |
|---|---|---|---|---|
| 总体 | 231 | 0.57 | 0.67 | 0.72 |
| single-hop | 114 | 0.53 | 0.63 | **0.67** |
| multi-hop | 43 | 0.35 | 0.49 | 0.65 |
| temporal | 63 | **0.84** | 0.91 | 0.91 |
| open-domain | 11 | 0.36 | 0.45 | 0.55 |

### 发现(关键,且和自建集结论冲突)

1. **自建集太乐观**:自建集说 direct/single 是 1.0、总体 0.79~0.90;真基准上 single-hop H@5 只有 0.53、总体 0.57。**自建集的 query 用词和记忆贴得太近、单人设、规模小,高估了真实能力。** 这条本身就是接基准最大的价值。
2. **双重瓶颈**:k 从 5→20,总体 H 仅升到 0.72 —— ~15% 是"召回到了但排 5 名外"(排序可救),**~28% 是前 20 都没召回到(根本 miss)**。单跳 H@20 仅 0.67 最扎眼。
3. **时序反而最强(0.84→0.91)**:LoCoMo 的时序题多带人名/事件等强锚点,证据 turn 易找;和自建集里"抽象换词时序"难的情形不同——说明短板高度依赖 query 形态。

### 对"是否及格"的修订结论

**以公开基准的检索召回看:单跳/多跳不及格**(单跳前 20 都漏 1/3)。此前基于自建集的"及格偏上"判断**下修**。注意:此为**检索召回**,**不等于** LoCoMo 排行榜的端到端问答分(那是 LLM-judge 打答案正确率,如 mem0 92.5),两者口径不同、不可直接比;且 turn 级检索(400 选 5)本就是偏难设定。

### 本轮未做优化(只做检验与记录)

避免 API 过度消耗,本轮只接基准、定位瓶颈,不仓促改算法。新增 `eval/locomo_eval.py`;外部数据集走 `.gitignore` 不入库。

### 保留版本

- 标记:`eval/memory-c3`。

### 下一轮候选(按对短板的针对性)

1. **检索粒度**(攻根本 miss):turn 级太碎,改 **session 级 / 滑窗 chunk 检索 + 命中后下钻到 turn**,通常能显著抬升 single-hop 召回——对标 mem0/LongMemEval 的做法。
2. **更强 embedding**:现用 `text-embedding-3-small`,换 `-large` 或 bge-m3 做对比。
3. **加一条朴素基线(BM25 / 纯 dense)** 在同一脚本上跑,给"0.57 算什么水平"一个本地参照(目前缺这个锚点)。
4. **硬化自建集**:换词更狠、加干扰项、多人设、扩到 150+,让自建分数可信。

## 第 2 轮 · 长期记忆:扩集 + 多跳 + 攻时序 — 2026-06-17

**数据集**:`eval/data/memory_zh.json` 扩到 **37 卡 / 52 query**;卡片带 `days_ago` 制造真实时间差;kind 新增 **multihop**(gold 为集合),并加 **Recall@5** 指标(多 gold 覆盖率)。

### 现状基线(c1 后,hybrid)与各桶短板

| 桶 | n | Hit@1 | Hit@3 | Recall@5 | MRR |
|---|---|---|---|---|---|
| direct | 14 | 1.00 | 1.00 | 1.00 | 1.00 |
| multihop | 10 | 0.90 | 1.00 | 0.78 | 0.95 |
| paraphrase | 14 | 0.43 | 0.86 | 0.86 | 0.63 |
| **temporal** | 14 | **0.64** | 0.71 | 0.86 | **0.71** |

→ **时序(temporal)是当前最弱项**,尤其 Hit@1:"最近 / 现在"类 query 没把"该时段的记忆"排到前面。

### 发现(含一条负向结论)

1. **时序的真正瓶颈是相关召回,不是纯排序**。诊断 base 融合排名发现:RRF 分数高度压缩(0.032/0.030/0.030…),相关度近乎噪声;像"我现在住在哪"的 gold(刚搬的新址)因不含"住"字,语义相关度排到 base#7,反被无关的近期卡盖过。
2. **宽窗 recency 重排是净负**:对全部候选按新鲜度加权,会把"近期但不相关"的卡(刚搬家、组会压力)顶上来,temporal 不升反降。→ 放弃宽窗方案。

### 优化(本轮采纳)

- **时态感知重排,但仅在已召回的 top_k 内**(`mybuddy/memory/long_term.py`:`_temporal_intent` + `_temporal_rerank`):
  - 识别 query 时态意图(`最近/现在/这几天/刚` → 偏新;`以前/之前/本科` → 偏旧);
  - 仅对最终 top_k 重排:`0.55·归一化相关度 + 0.45·recency`,**不改召回集合(Recall@k 不变)**,把伤害限制在"重排已展示项",规避宽窗的副作用。
- 同时修正一处过严 gold("最近不开心的事"也应接受"组会压力大")。

### 隔离 A/B(同一份嵌入,只切重排开关 → 纯归因)

| 桶 | 关重排 MRR / Hit@1 | 开重排 MRR / Hit@1 | Δ |
|---|---|---|---|
| **temporal** | 0.782 / 0.714 | **0.818 / 0.786** | **+0.036 / +0.072** |
| paraphrase | 0.631 / 0.429 | 0.667 / 0.500 | +0.036 / +0.071 |
| direct | 1.000 / 1.000 | 1.000 / 1.000 | 0(无回归) |
| multihop | 0.950 / 0.900 | 0.950 / 0.900 | 0(无回归) |
| 总体 | 0.832 / 0.750 | 0.852 / 0.788 | +0.020 / +0.038 |

→ top-k 门控重排:temporal Hit@1 **+0.072**、MRR **+0.036**,direct/multihop **零回归**。改幅温和但干净、安全。236 测试全绿。

### 保留版本

- 标记:`eval/memory-c2`。

### 仍存在的短板(下一轮候选)

- **抽象短 query 的相关召回**才是时序天花板:gold 常根本不在 top-k 候选里,重排无能为力。下一轮真正的杠杆是**召回相关度**本身——换更强 embedding(当前 `text-embedding-3-small`)、或为时间/实体建结构化索引(对标 mem0 的图谱 / LongMemEval 的 time-aware indexing)。
- multihop 的 **Recall@5 仅 0.78**:多约束 query 还差一张卡没召全,可加"多子查询分解 + 合并"。
- paraphrase Hit@1 仍 0.5,有提升空间。

**数据集**:`eval/data/memory_zh.json`(22 张记忆卡 / 20 条 query;kind = direct 9 / paraphrase 9 / temporal 2)。
**评测脚本**:`uv run python eval/memory_eval.py --mode both`。
**口径**:top_k=5;hybrid 用 OpenRouter `text-embedding-3-small` + RRF(k=60)。

### 基线 vs 优化(同一数据集)

| 桶 | 指标 | 词法基线 | 词法+语义RRF | Δ |
|---|---|---|---|---|
| 总体 | MRR | 0.617 | **0.767** | **+0.150** |
| 总体 | Hit@3 | 0.70 | **0.85** | +0.15 |
| direct | MRR | 0.944 | 1.000 | +0.056 |
| **paraphrase(换词)** | MRR | 0.315 | **0.593** | **+0.278** |
| paraphrase | Hit@3 | 0.444 | **0.778** | +0.334 |
| temporal | MRR | 0.500 | 0.500 | 0 |

### 发现

1. **词法召回在"换词类"query 上明显塌陷**(MRR 0.315):用户换个说法问(如"我能吃虾吗"对应"海鲜过敏")就召回不到。语义 RRF 能补回大半,换词类 MRR 近翻倍。
2. **生产代码没真正用上已配置的语义层**:`config.yaml` 里 `embedding.enabled: true`,自动注入路径(`manager.py`)有传 `use_semantic`,但**`recall_memory` 工具**(用户显式问"你还记得…"时走的路径)调 `ltm.search` 时漏传,跑的是纯词法——等于把上面的 +0.28 白白丢掉。

### 优化(本轮采纳)

- `mybuddy/tools/memory_tool.py`:`recall_memory` 的 `ltm.search(...)` 加 `use_semantic=True`,让显式召回也走词法+语义 RRF 融合。
- 影响面:236 测试全绿;召回质量按上表提升。

### 保留版本

- 标记:`eval/memory-c1`(本轮优化生效后的快照)。

### 仍存在的短板(下一轮候选)

- **时序推理持平(MRR 0.5)**:语义召回对"这周三我有哪些安排"这类时间约束帮助有限——缺显式时序过滤 / 推理。这与公开 benchmark(LoCoMo/LongMemEval)最拉差距的维度一致,是记忆维度后续优化的重点。
- 数据集偏小(20 条),后续可扩到 50+ 并加入多跳题。
