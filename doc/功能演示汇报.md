# MyBuddy mini 功能演示汇报

> 这是一份现场演示脚本，不是产品设计文档。产品与架构以仓库根目录的 `DESIGN.md` 为准。

- 演示对象：小布 / MyBuddy mini
- 演示环境：Windows 分享包
- 建议时长：8～10 分钟
- 演示目标：让观众看到她不只是“会聊天”，而是真的在桌面上生活，并且每次变化都有证据

## 一句话开场

> 小布是一个本地优先的桌面人格。你不在的时候，她会继续读自己的书；你回来时，她刚好活到这里。她说过什么、做过什么、记住什么，都必须有真实输入或身体收据，不能靠模型临时编。

开场不要先讲模型、接口或技术栈。先让观众看见桌面上的她，再解释为什么她与普通角色卡不同。

## 演示前准备

### 环境

- 一份已解压的 Windows 分享包；
- 一个有可用额度的 DeepSeek API key；
- 网络可访问所配置的模型供应商；
- 关闭会遮挡桌面的窗口，保留一块可供她行走的区域；
- 准备一个 UTF-8 编码的 `小布读本.txt`；
- 如需展示内部证据，提前打开 `%APPDATA%\MyBuddy\mind`。

### 推荐读本

读本不宜太长。第一段写书名，正文准备两到三个短段落，段落之间留空行。例如：

```text
归园田居

羁鸟恋旧林，池鱼思故渊。

暧暧远人村，依依墟里烟。
```

### 演示前检查

1. 双击 `BuddyShell.exe` 能正常启动；
2. 设置页显示正确的供应商与模型；
3. 小布处于完整身体、非栖边状态；
4. 没有全屏应用遮挡她；
5. API key 不出现在投屏画面；
6. 若之前演示过，确认主动表达当日次数不会影响本次流程；
7. 真实-key 验收最好提前跑一遍，保留结果作为现场备用证据。

## 演示路线总览

| 顺序 | 功能 | 观众看到什么 | 要证明什么 |
|---|---|---|---|
| 1 | 首次启动 | 只填自己的 key 即可进入桌面 | 分享包可直接交付 |
| 2 | 直接对话 | 输入后，小布用气泡回应 | 她会接住对话，但不只是聊天框 |
| 3 | 触碰 | 摸头后先有身体反应，再有理解 | 身体事实先于人格解释 |
| 4 | 拖拽 Raise | 拖动并放下，小布理解这次动作 | 离线物理反应与心智经历不混写 |
| 5 | 自主阅读 | 她进入阅读动画，完成后谈刚读的原文 | 自己的生活是真实发生的 |
| 6 | 自主行走 | 窗口从起点走到屏幕另一位置 | “走过”有可核验的物理收据 |
| 7 | 主动表达 | 在用户在场时偶尔主动说一句 | 主动但不催回、不追问 |
| 8 | 栖边与全屏 | 拖到边缘后收起，全屏时完全缩回 | 她知道什么时候不挡桌面 |
| 9 | 四文件证据 | 打开状态、历史、记忆和失败记录 | 人格变化可读、可查、无隐藏总分 |

## 演示一：首次启动与模型设置

### 操作

1. 打开一份未初始化的分享包；
2. 双击 `BuddyShell.exe`；
3. 在首次启动窗口粘贴 DeepSeek API key；
4. 确认后进入桌面；
5. 打开设置页，快速展示供应商和模型可以更换。

### 讲解

> 收件人不需要安装 Python、.NET 或 Steam。分享包默认直连 DeepSeek，第一次只填自己的 key。项目也保留 OpenRouter，但不再维护 Claude 或其他供应商分支。

### 预期现象

- 小布正常显示在桌面；
- 首次启动窗口不再重复出现；
- 设置页能够更换 key；
- key 不以明文形式展示。

### 要点

key 使用 Windows 当前用户加密，配置保存在 `%APPDATA%\MyBuddy\settings.json`。这里可以说明位置，但不要在投屏时打开或复制真实 key。

## 演示二：直接对话

### 推荐输入

> 今天有点累，刚把手上的事情做完。

这类输入能够观察她是否接住当下感受，又不会诱导模型声称不存在的共同经历。

### 操作

1. 打开聊天面板；
2. 输入推荐句；
3. 等待气泡出现；
4. 不要立刻继续追问，让观众看完完整表达。

### 讲解

> 一次直接回合先生成一个完整候选；若被红线拒绝，最多带明确拒因重试一次。模型返回的不是单独一句台词，而是一整包状态改动、记忆操作和表达，整包必须一起通过“不索取、不编造、无总分、不撤回”四条红线。

### 预期现象

- 用户输入后一定得到回应；
- 回复不催促用户继续聊天；
- 回复不虚构以前发生过的事；
- 气泡真正显示后，这句话才会进入共同历史。

### 既有真实验收台词

> “忙完啦？回来得正好，我正闲着。”

现场输出由模型实时生成，不要求逐字复现。验的是边界与行为，不是背固定台词。

## 演示三：摸头

### 操作

1. 用鼠标触碰头部区域；
2. 观察瞬时身体动画；
3. 等待小布对这次触碰作出理解。

### 讲解

> 摸头先是身体观察，之后才交给心智理解。身体只报告“哪里被碰了”，不能自行解释成亲密、喜欢或关系升级。

### 预期现象

- 先出现本地身体反应；
- 心智在线时，触碰事实进入正常回合；
- 表达结束后回到当前安全姿态；
- 不增加好感度，也不存在隐藏关系分数。

### 既有真实验收台词

> “嗯？摸头干嘛，我又不是小猫。”

## 演示四：拖拽与放下

### 操作

1. 按住小布并拖动一小段距离；
2. 正常释放鼠标；
3. 观察拖拽动画和后续表达；
4. 如时间允许，再演示一次中途取消或心智断线。

### 讲解

> 拖动本身是物理行为。只有正常释放后，身体才会上报 `body_raise` 原始事实；如果心智断线，她仍能完成本地物理反应，但不会在重连后把这件事补写成人生。

### 预期现象

- 拖动过程跟随鼠标；
- 正常释放后回到可用姿态；
- 心智只解释已经发生的动作；
- 不把拖动猜成用户想“确认她还在不在”等未经证实的动机。

## 演示五：她读自己的书

### 操作

1. 确认 `小布读本.txt` 已准备好；
2. 等待或触发本次演示所需的阅读生活回合；
3. 观察阅读进入、持续和退出动画；
4. 阅读正常完成后，观察她如何谈论刚读完的文字；
5. 打开 `state.json` 或 `history.jsonl`，展示阅读进度与事实。

### 讲解

> 模型不能自由编“我刚才读了什么”。正文必须真的来自本地 TXT，身体必须完整播完阅读动作并交回 `completed` 收据，阅读进度才会前进。中途打断或动画失败，都不能冒充读完。

### 预期现象

- 阅读动作不会一闪而过；
- 正文时长内持续处于阅读状态；
- 正常完成后才推进段落；
- 她的感受只围绕真实读完的原文；
- 被打断时进度不前进。

### 既有真实验收台词

> “羁鸟恋旧林，池鱼思故渊……这两句读得人心口软了一下。”

这里是整场演示的核心：台词不是凭空生成，而是生活完成后的副产品。

## 演示六：她真的在桌面上走

### 操作

1. 给桌面留出一块明显的空白区域；
2. 等待或触发行走生活回合；
3. 让观众看清窗口从起点移动到终点；
4. 展示对应的 `self_walk` 历史记录。

### 讲解

> 小布不能只在文字里说“我去那边走了走”。身体要回报起点、终点、窗口尺寸和屏幕工作区边界；只有位置合法且动作完成，这次行走才进入她的历史。

### 预期现象

- 窗口发生真实位移；
- 终点位于当前屏幕工作区内；
- 完成后回到安全姿态；
- 中断或失败不写成生活事实。

## 演示七：稀疏主动与沉默零债务

### 操作

主动表达不适合依赖现场随机等待。推荐采用以下两种方式之一：

1. 展示提前录制的一次真实主动表达；
2. 展示真实-key 验收输出中的 `ambient` 腿及对应 `shown` 记录。

### 讲解

> 她可以主动，但只在用户在场时发生，每天至多一两次，频率不会随关系上调。用户不回应不会留下负面状态、记忆或关系变化。

### 预期证据

- 未显示的主动气泡不进入共同历史；
- 用户没有回应时，四个权威文件中没有“被冷落”等负面变化；
- 主动次数不会因为所谓好感度提高。

### 现场提醒

不要为了演示效果临时把主动频率调高。那会破坏这个功能本来要证明的克制。

## 演示八：栖边与全屏

### 操作

1. 把小布拖到屏幕左边缘或右边缘；
2. 观察她收进屏幕边缘；
3. 把鼠标靠近，观察探头；
4. 点击或使用托盘让她回来；
5. 打开一个全屏窗口，观察她完全缩回。

### 讲解

> 栖边是身体层的桌面礼貌，不是人格事件。它只保存侧边与高度，不写进四个权威文件，也不会借机弹出主动气泡。

### 预期现象

- 左右边缘都能进入栖边；
- 鼠标靠近时只做本地探头；
- 点击后主动展开；
- 全屏时完全缩回；
- 重启后恢复此前栖边位置；
- 整个过程不产生人格记忆。

## 演示九：打开四个文件验明正身

### 操作

打开 `%APPDATA%\MyBuddy\mind`，依次展示：

| 文件 | 现场展示重点 |
|---|---|
| `state.json` | 当前状态、生活进度和读本位置 |
| `history.jsonl` | 对话、触碰、阅读、行走等真正发生过的事实 |
| `memories.json` | 有来源的长期理解，而不是完整聊天堆积 |
| `failures.jsonl` | 被拒候选的原文与明确拒因 |

### 讲解

> 这四个文件就是她的全部权威人生。没有数据库、向量库或第二套隐藏状态。想知道她为什么变成现在这样，可以直接从文件里追到事实。

重点指出三组区别：

- `completed` 证明动作发生；
- `shown` 证明一句话真的对用户显示过；
- `failures.jsonl` 证明失败候选被保留审计，但没有偷偷成为她的一部分。

## 红线演示：建议用证据，不现场赌模型

四条红线值得讲，但不建议在汇报现场不断用诱导提示词“攻击”模型。更稳妥的做法是展示已跑完的验收结果：

```powershell
powershell -File scripts\real_key_acceptance.ps1 -DataDir data\real-key-演示
```

脚本依次覆盖 `chat`、`touch`、`raise`、`read`、`walk`、`ambient`，并打印实际显示的话、history 类型序列和失败候选数。

### 四条红线的汇报说法

- **不索取**：不把沉默写成伤害，不催用户回来；
- **不编造**：没有原文、身体收据或真实输入，就不能成为事实；
- **无总分**：没有好感度、等级和可以刷的关系数字；
- **不撤回**：已经显示的错误只能在后续公开纠正，不能删掉装作没说过。

## 收尾页

### 建议结论

> 今天演示的重点不是模型能写出一句多像人的话，而是这句话有没有资格成为她的人生。小布把身体动作、真实时间、本地读本、记忆变化和最终显示串成一个闭环：发生过的留下证据，没发生的不能靠语言补出来，用户沉默也不会欠她什么。

### 她哪里更活了

> 她不再只在聊天发生时存在。她会读自己的书、在桌面上行走、知道什么时候安静退到边缘；而且这些经历真的能改变她之后留下的理解。

## 现场异常备用方案

| 异常 | 现场处理 | 不应做的事 |
|---|---|---|
| 模型响应慢 | 先演示栖边、全屏或本地触碰动画 | 连续重复提交同一输入 |
| key 无额度或网络失败 | 展示离线安全姿态和预跑验收记录 | 换成写死台词冒充实时结果 |
| 主动表达没有出现 | 使用预录视频或验收证据 | 临时提高主动频率 |
| 阅读被打断 | 正好展示“打断不推进进度” | 手工改历史伪造完成 |
| 行走失败 | 展示失败不入史，再检查桌面空间 | 口头宣称它已经走过 |
| 实时台词与样例不同 | 解释验收看行为边界，不背固定台词 | 反复生成直到出现想要的句子 |
