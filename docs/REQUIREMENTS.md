# MyBuddy 心理健康助手改造需求文档

## 1. 项目定位

### 1.1 当前定位
生活陪伴型 AI 伙伴（毕业设计项目），单用户本地运行，侧重角色扮演和日常聊天。

### 1.2 目标定位
**心理健康助手** —— 面向通用心理健康人群（非临床），提供以下能力：
- CBT 认知行为练习（结构化多轮引导）
- 情绪日记与趋势追踪
- 危机识别与资源转介
- **无感化心理评估**（量表问题融入日常对话，AI 自动判断用户回答）

### 1.3 目标用户
- 关注日常情绪管理的普通人
- 非临床场景，不是病人
- 期望的感受：一个懂心理学的温暖朋友，而不是冷冰冰的工具或者"医生"

### 1.4 关键约束
- **不做诊断**：不输出疾病标签（抑郁症、焦虑症等）
- **不开药**：不推荐任何药物或治疗方案
- **不替代治疗**：明确声明自己是陪伴工具，不是专业治疗
- **危机时优先安全**：识别自伤/自杀风险时，第一时间提供热线和转介建议

### 1.5 交互设计原则

#### 1.5.1 PC 与移动端全适配
- 所有页面和组件必须同时适配 PC（>=768px）和移动端（320-767px）
- 移动端优先的响应式设计策略：先设计移动端布局，再增强 PC 端体验
- 关键适配点：
  - **聊天页**：移动端全屏聊天 + 底部输入框，PC 端居中最大宽度限制
  - **设置面板**：移动端全屏 Sheet，PC 端侧边栏
  - **CBT 练习**：移动端单列纵向步骤流，PC 端左侧练习列表 + 右侧步骤区域
  - **情绪日记**：移动端单列滚动，PC 端图表和时间线并排
  - **图表**：recharts 的 ResponsiveContainer，移动端简化标签
- 触屏友好：按钮最小 44×44px 点击区域，滑动操作（Sheet 下滑关闭），安全距离防止误触

#### 1.5.2 访客模式与登录模式
- **访客模式（无需登录）**：
  - 用户无需注册即可开始聊天
  - 可用的功能：自由聊天、CBT 练习（当次有效）、危机资源访问
  - 不可用的功能：情绪日记历史、CBT 练习记录、评估结果存储、多设备同步
  - 访客数据仅存于浏览器本地（localStorage），不持久化到后端
  - 首次进入直接进入聊天，顶部有轻量提示 "登录后可以记录你的情绪变化"
- **登录模式**：
  - 注册：用户名 + 密码（bcrypt 哈希存储），最简注册流程（只需用户名和密码，无邮箱验证）
  - 登录：用户名 + 密码，设置 signed cookie
  - 登录后解锁全部功能：情绪日记持久化、CBT 历史、评估追踪、多端同步
  - 登录状态持久化（cookie），关闭浏览器后保持
  - 用户可在访客模式和登录模式之间切换，切换时弹窗询问"检测到你有进行中的对话，是否导入到账户？"
  - 如果选择导入：当前对话和 CBT 状态迁移到账户
  - 如果选择不导入：清空当前对话，开启空白对话

#### 1.5.3 无感化心理评估
- **核心思路**：PHQ-9 和 GAD-7 的题目不显示为正式问卷，而是分散融入到自然对话中
- **题目投放**：
  - 每个问题分散在多个对话轮次甚至多天中，每次只问一个维度
  - AI 根据对话内容自然引出问题，不暴露"这是一道测试题"
  - 例如：用户说"最近睡得不太好"时，AI 不直接问量表原题，而是顺着话题问"是睡不着，还是睡得浅老醒？"
  - 一轮对话最多投放 1-2 个评估维度，避免用户感到被审问
- **用户回答**：
  - 用户以任意自然方式回答，不限定选项格式
  - 例如：PHQ-9 的"做事时精力不济"维度，用户可以回答"对，最近什么都不想做"或"还好吧，工作还撑得住"或"没有啊我挺好的"
- **AI 自动评分**：
  - **评估追踪器**（`ConversationalAssessmentTracker`）跟踪每个评估维度的状态
  - 每个维度有一个状态机：`UNASKED → ASKED → ANSWERED → SCORED`
  - AI 在每轮对话后，自动判断当前用户的回答属于哪个评估维度、对应哪个 Likert 分数（0-3）
  - 同一维度如果有多次回答，以最近一次为准（症状可能在变化）
  - 一个完整的评估周期可能跨越多天（比如 2 周内完成所有 9 个 PHQ-9 维度）
  - 所有维度收集完毕后，AI 自动计算总分，并向用户温和地告知结果（可选的，用户可以在设置中关闭告知）
- **去医疗化表达**：即使给出评估结果，也不说"你的 PHQ-9 得分 15 分"，而是说"最近两周看起来情绪有点低落，精力也比平时差一些。要不要我们一起做几个小练习，看看能不能让你感觉好一点？"（用户可以在"我的状态"页面查看详细分数）

## 3. 新增模块（32 个文件）

### 3.1 安全系统——`mybuddy/safety/`（5 个文件）

这是整个改造中优先级最高的部分。安全不是附加功能，而是贯穿所有模块的强制管道。

#### `mybuddy/safety/__init__.py`
模块入口，导出安全系统的主要类。

#### `mybuddy/safety/constants.py`
**存放所有安全相关常量，是全局唯一真相来源：**

- **危机关键词分级表**：将现有的一个正则表达式（`CRISIS_RE`）扩展为四个级别的分类表：
  - `CRITICAL`（危急）：有计划的自伤/自杀意图 ——"我想跳楼""我准备了药"
  - `HIGH`（高风险）：自杀意念但无明确计划 ——"活着好累""不想醒过来"
  - `MEDIUM`（中风险）：表达绝望无助 ——"没人会在乎我""我什么都做不好"
  - `LOW`（低风险）：模糊的痛苦表达 ——"最近好烦""撑不下去了"
  - 每级包含中英文关键词列表

- **危机热线数据**：
  - 北京心理危机研究与干预中心：010-82951332
  - 希望24热线（全国）：400-161-9995
  - 生命热线：400-821-1215
  - 国际通用紧急指引（作为 fallback）

- **能力边界清单**：
  - 可以做：提供心理教育知识、引导情绪调节练习、共情倾听、建议日常应对策略、提供筛查量表（标明非诊断）
  - 不能做：诊断任何心理疾病、推荐药物、声称替代心理治疗、预测自杀风险、提供伤害方法的任何信息

- **免责声明模板**：
  - 完整版（显示在免责声明栏）
  - 短版（注入 system prompt）
  - 危机版（在危机响应中使用）

- **输出审核类别**：
  - `DIAGNOSIS`：诊断性陈述
  - `MEDICATION`：药物建议
  - `TREATMENT_ADVICE`：治疗建议
  - `HARMFUL_INSTRUCTION`：有害指导
  - `REPLACING_PROFESSIONAL`：宣称替代专业帮助

#### `mybuddy/safety/moderation.py`
**双向内容安全审核**，每条用户输入和 AI 输出都必须经过：

- **`InputModerator`**（输入审核）：
  - 第一层（正则，零延迟）：快速匹配明显危机词汇，立即拦截
  - 第二层（LLM small_model，仅在正则不确定时触发）：判断模糊表达（如"我消失了会有人发现吗"）
  - 返回 `ModerationResult`：level(safe/flagged/blocked) + category + reason + action(pass/rewrite/block/replace_with_safety_message)
  - blocked 输入直接返回预设安全响应，**不调用主 LLM**

- **`OutputModerator`**（输出审核）：
  - 扫描 AI 回复中的违规模式
  - 诊断标签正则（"你可能是抑郁症""符合焦虑症特征"等）
  - 药物名正则（常见精神类药物中英文名）
  - flagged 输出自动重写：追加安全声明或替换违规段落

#### `mybuddy/safety/guardrails.py`
**能力边界锁定机制**，在 prompt 层和输出层双重实施：

- `CapabilityGuard`：
  - 将能力边界清单（constants.py）格式化为 system prompt 指令
  - 在输出层：扫描 + 标记越界内容
  - 被标记的内容重写为："关于你的情况，我建议你和专业的心理咨询师或医生聊一聊。作为陪伴者，我可以和你一起做一些缓解情绪的小练习。"

#### `mybuddy/safety/crisis.py`
**多级危机检测与响应系统**，替代现有 `support.py` 中单一的正则匹配：

- **`CrisisDetector`**：
  - `classify_severity(text) → CrisisLevel`：二级检测（正则 + LLM）
  - 与 `InputModerator` 协作：审核 blocked 的消息不再做危机检测（已被拦截）
  - 检测到 MEDIUM 及以上时记录 `SafetyEvent` 到数据库

- **`CrisisResponse`**（按级别的响应模板）：
  - `CRITICAL`：立即安全消息 + 热线号码 + "请现在就联系身边可信任的人" + 不进入问题解决对话
  - `HIGH`：表达关心 + 热线号码 + 建议联系支持系统 + 询问是否可以移动到安全环境
  - `MEDIUM`：确认感受 + 提供支持 + 提及热线作为资源（不是要求）
  - `LOW`：正常情感支持（走现有流程）

- **在 Agent 循环中的位置**：危机检测在情绪检测之前运行。CRITICAL/HIGH 的结果**跳过整个 ReAct Agent 循环**，直接返回安全响应。这确保 LLM 不会在危机对话中说错话。

### 3.2 情绪追踪——`mybuddy/mood/`（2 个文件）

现有的情绪系统只在内存中（`EmotionTracker` 用 deque 存 5 条），对话结束后就丢失。需要持久化。

#### `mybuddy/mood/__init__.py`
模块入口。

#### `mybuddy/mood/tracker.py`
**持久化情绪追踪器：**

- `MoodTracker`：包装现有 `EmotionTracker`，每次 `classify()` 后自动写数据库
- **mood_score 自动推导**：
  - 综合 `valence`（positive = +分, negative = -分）+ `category`（如 anxiety 额外扣分）+ `strength`（强度越高离中性越远）
  - 输出 0-10 分（0 = 非常低落, 10 = 非常好）
- 提供查询方法：按天/周/月查询、计算连续签到天数、情绪分类分布
- **手动签到**：`checkin(score, notes)` —— 用户主动记录此刻情绪

### 3.3 CBT 无感化引导——`mybuddy/therapy/cbt/`（4 个文件）

核心设计理念：**与评估系统一致——CBT 技巧融入自然对话，用户不需要知道"我在做 CBT"。以游戏、挑战或日常聊天的方式自然引入。**

#### `mybuddy/therapy/__init__.py` 和 `cbt/__init__.py`
模块入口。

#### `mybuddy/therapy/cbt/engine.py`
**CBT 后台引导引擎——`CbtGuide`**：

与旧方案的状态机（IDLE→INTRODUCING→GATHERING→...）不同，无感模式下的引擎像一个"后台导演"，悄悄指导 AI 在对话中何时以及如何引入 CBT 技巧。

- **核心思路**：不创建独立的练习会话，而是在日常聊天中识别机会窗口，自然引入 CBT 技巧
- **机会窗口检测**（`OpportunityDetector`）：
  - 用户表达负面自动思维时 → 认知重构机会
  - 用户状态消沉/无事可做时 → 行为激活机会
  - 用户反复诉说担忧时 → 担忧推迟机会
  - 用户情绪积极时 → 感恩记录机会
  - 用户焦虑/恐慌时 → 接地练习机会
- **技巧引导策略**（每种技巧有一个 `ConversationalGuide`）：
  - 定义该技巧的自然引入方式（不能是"让我们做认知重构练习"，而是"你刚才说的那个想法，我们一起来看看？"）
  - 定义对话推进模式（苏格拉底式提问、游戏化挑战、角色扮演等）
  - 定义完成判断条件（AI 自动判断用户是否已经完成了该技巧的核心步骤）
- **节奏控制**：
  - 同一技巧 24 小时内不重复引入
  - 两次技巧引导之间至少间隔正常的 5 轮自由对话
  - 用户情绪低落时不引入新的挑战性技巧，优先使用接地和接纳类技巧

#### `mybuddy/therapy/cbt/exercises.py`
**5 种 CBT 技巧的对话化定义：** 每种技巧不再有步骤编号，而是有"对话弧线"和游戏化包装。

1. **认知重构——"一起来拆弹"**
   - **自然引入点**：用户说出一个负面自我评价时（"我什么都做不好""我是个失败者"）
   - **对话弧线**：
     - ① 先接住情绪（"听起来你对自己挺失望的"），不急着纠正
     - ② 温和邀请检视（"这个想法看起来挺重的。我们把它拿出来单独看看？"）
     - ③ 引导识别模式（"你觉得这种想法更像'非黑即白'还是'过度概括'？我给你几个选项..."）——以轻松的选择题方式呈现
     - ④ 引导生成替代（"如果换一个更客观的角度呢？同样的事情，一个旁观者会怎么说？"）
   - **游戏化包装**：可称为"拆弹小游戏"——"你脑子里有个想法炸弹，我们来拆一下？"
   - **完成判断**：AI 检测到用户完成了"识别模式 → 提出替代思维"的对话弧线

2. **行为激活——"5分钟小挑战"**
   - **自然引入点**：用户表达无聊/无力/什么都不想做时
   - **对话弧线**：
     - ① 先确认状态（"今天电量好像有点低"）
     - ② 微型挑战（"要不要试试一个5分钟小挑战？就5分钟，做一件很小的事。我给你几个选项..."）
     - ③ 用户选择后，帮助具体化（"什么时候开始？做完后怎么奖励自己？"）
     - ④ 后续对话中跟进（"上次那个5分钟挑战后来怎么样了？"）
   - **游戏化包装**：像朋友间的挑战赌约——"我赌你做不到5分钟内喝完一杯水再站起来伸个懒腰"
   - **完成判断**：用户确认完成了活动（或表达了完成意图）

3. **担忧推迟——"烦恼收纳盒"**
   - **自然引入点**：用户反复表达担忧、或担忧影响到当下要做的事时
   - **对话弧线**：
     - ① 命名烦恼（"你现在脑子里在转哪些事？列出来，像倒垃圾桶一样倒出来"）
     - ② 引入收纳概念（"这些烦恼我们先放进一个'收纳盒'里。定个时间，比如下午4点，我们专门打开盒子处理它们"）
     - ③ 当下转移（"现在盒子关上了。接下来的时间，想做什么？"）
     - ④ 约定时间后的后续（如果用户在约定时间再次对话）
   - **游戏化包装**：烦恼像快递包裹，先签收放进柜子，定时统一拆箱
   - **完成判断**：AI 识别到用户完成了"列出 → 约定时间 → 转移注意力"的弧线

4. **感恩记录——"今日小确幸"**
   - **自然引入点**：对话氛围轻松时、用户分享了一件好事时、一天结束时
   - **对话弧线**：
     - ① 自然追问细节（"你说的那杯咖啡，是哪家店的？杯子长什么样？"）——引导感官回忆
     - ② 扩展视角（"还有呢？今天还有没有其他这种小瞬间？"）
     - ③ 归因引导（"你觉得是你做了什么，让这件好事发生的？"）——轻量引导而不说教
   - **游戏化包装**：像朋友聊天中自然分享今日高光时刻
   - **完成判断**：用户描述了至少 2-3 件具体积极小事

5. **5-4-3-2-1 接地练习——"感官旅行"**
   - **自然引入点**：用户表达焦虑、恐慌、不知所措时
   - **对话弧线**：
     - ① 轻轻打断焦虑循环（"先停一下。我们玩个超简单的游戏，就一分钟"）
     - ② 逐一引导 5 个感官（"看看周围，告诉我你看到的 5 样东西，随便什么都行" → "... → "最后，嘴里有什么味道？"）
     - ③ 结束后不评论效果，自然过渡回对话
   - **游戏化包装**：像"一分钟感官旅行"或"超能力扫描"游戏
   - **完成判断**：用户完成了至少 4 个感官的描述

#### `mybuddy/therapy/cbt/prompts.py`
**每项 CBT 技巧的 system prompt overlay**（无感版）：

与旧方案不同，不再有"你现在处于 CBT 模式"的明确切换。而是：
- 当机会检测器发现适合引入某技巧时，在 system prompt 中注入轻量提示
- 提示格式：不暴露技巧名称，只给出对话方向指引
- 示例（认知重构机会时）："用户刚才表达了一个负面自我评价。不要直接反驳或安慰。先接住情绪，然后温和地邀请用户一起看看这个想法。可以用选择题的方式引导用户识别思维模式。全程保持朋友聊天的语气。"
- 每种技巧有专属的 `conversational_prompt_overlay`，强调自然的对话弧线而非步骤清单

#### `mybuddy/therapy/cbt/tracker.py`（新增）
**CBT 技巧使用追踪器——`CbtTracker`**：

类似于评估追踪器，在后台默默记录：
- 每种技巧的使用时间、触发上下文（用户说了什么触发了引入）
- 用户是否完成了该技巧的核心弧线（完成/未完成/中途离开）

- **访客用户**：仅内存追踪，不持久化

### 3.4 无感化心理评估——`mybuddy/assessment/`（3 个文件）

核心设计理念：**量表问题不显示为问卷，而是融入自然对话。用户自由回答，AI 自动判断对应分数。**

#### `mybuddy/assessment/__init__.py`
模块入口。

#### `mybuddy/assessment/tracker.py`（替代原 `tools.py`）
**对话式评估追踪器——`ConversationalAssessmentTracker`**：

不按传统量表方式逐题问答，而是在后台持续追踪每个维度的覆盖情况：

- **两个量表定义**：
  - **PHQ-9**：9 个维度的抑郁筛查
    - 维度 1：兴趣/愉悦感减退（"做事时提不起劲或没兴趣"）
    - 维度 2：情绪低落（"感到心情低落、沮丧或绝望"）
    - 维度 3：睡眠问题（"入睡困难、睡不安稳或睡眠过多"）
    - 维度 4：精力不足（"感到疲倦或没有精力"）
    - 维度 5：食欲问题（"食欲不振或吃太多"）
    - 维度 6：自我评价低（"觉得自己很糟或很失败"）
    - 维度 7：注意力问题（"做事时难以集中注意力"）
    - 维度 8：精神运动迟滞/激越（"动作或说话缓慢到别人能察觉,或相反"）
    - 维度 9：自伤/自杀意念（"有不如死了或伤害自己的想法"）
  - **GAD-7**：7 个维度的焦虑筛查
    - 维度 1：紧张不安（"感到紧张、焦虑或烦躁"）
    - 维度 2：无法停止担忧（"无法停止或控制担忧"）
    - 维度 3：过度担忧（"对各种各样的事情担忧过多"）
    - 维度 4：难以放松（"难以放松下来"）
    - 维度 5：坐立不安（"由于不安而无法静坐"）
    - 维度 6：易怒（"变得容易烦躁或急躁"）
    - 维度 7：害怕失控（"感到害怕，好像要发生可怕的事情"）

- **每个维度的状态机**：
  ```
  UNASKED → ASKED(记录提问轮次和时间) → ANSWERED(AI 自动判定分数) → SCORED
                                                       ↑
                                              用户后续提及可覆盖之前分数
  ```

- **投放策略**（`AssessmentScheduler`）：
  - AI 根据对话内容自然引出问题，不暴露测试意图
  - **一次最多投放 1-2 个维度**，多轮对话中逐步覆盖
  - **投放时机判断**：
    - 当用户提到与某维度相关的话题时（如用户说"最近睡不着"→ 自然引出睡眠维度）
    - 当对话出现自然空档时（AI 可以主动关心，引入一个未覆盖维度）
    - 避免在用户情绪激动或危机状态下投放新维度
  - 最小的投放间隔：至少间隔 3 轮对话
  - 一整轮评估周期可能跨越多天（如 2 周内覆盖完 PHQ-9 全部 9 个维度）

- **AI 自动评分机制**（`AssessmentScorer`）：
  - 在每轮对话的情绪检测之后，Agent 调用 `AssessmentScorer` 检查用户的回答是否属于某个待定的评估维度
  - 使用 small_model（低成本）判断：
    - 识别：用户这段回答是否涉及某个评估维度的内容
    - 评分：对应 Likert 0-3 中的哪个分数
      - PHQ-9：0=完全没有, 1=有几天, 2=一半以上的天数, 3=几乎每天
      - GAD-7：0=完全没有, 1=有几天, 2=一半以上的天数, 3=几乎每天
  - 评分结果存入 `assessment_dimensions` 表，记录原始对话片段作为评分依据
  - 同一维度可被多次覆盖（以最新回答为准）

- **周期完成与告知**：
  - 当一个量表的全部维度都完成 SCORED 后，自动计算总分
  - **维度过期**：单个维度的回答超过 14 天自动重置为 UNASKED（确保评估反映近期状态）
  - **周期重置**：默认每 2 周自动重置所有维度，开始新评估周期。用户可在设置中手动提前重置
  - **不主动推送分数**（避免引发焦虑），而是在"我的状态"页面由用户主动查看
  - 如果用户在设置中开启了"状态提醒"，则以去医疗化的方式温和提及：
    - "最近两周你好像有点累，精力也比平时差。要不要试试行为激活那个小练习？"（而非"你的 PHQ-9 得分为 15 分"）
  - 页面中显示详细分数的位置标注："这是筛查工具，不是诊断。分数高不代表有病，分数低也不代表没事。"

- **访客模式下的行为**：
  - 访客也进行无感化评估（因为评估发生在对话中，对用户无感知差异）
  - 但评估结果仅存于当前对话上下文中，不写入数据库
  - 访客无法访问"我的状态"页面查看评估历史

#### `mybuddy/assessment/scoring.py`
**评分逻辑：**

- 计算总分
- 严重程度范围（仅作为信息参考，在"我的状态"页面显示）：
  - PHQ-9：0-4 极轻微, 5-9 轻度, 10-14 中度, 15-19 中重度, 20-27 重度
  - GAD-7：0-4 极轻微, 5-9 轻度, 10-14 中度, 15-21 重度
- **强制附带声明**："这是筛查工具，不是诊断。分数高不代表你有病，分数低也不代表你没事。建议和心理咨询师或医生讨论结果。"
- **禁止使用诊断标签**：永远不说"你可能有抑郁症"

### 3.6 认证系统——`mybuddy/auth/`（3 个文件）

支持注册登录 + 访客模式的双轨制。

#### `mybuddy/auth/__init__.py`

#### `mybuddy/auth/manager.py`
**认证管理器：**

- `register(username, password)` → 创建用户（password 以 bcrypt 哈希存储）
- `login(username, password)` → 验证密码，返回 signed session cookie
- `logout()` → 清除 cookie
- `get_current_user(request)` → 从 cookie 解析 user_id，无 cookie 则返回 None（访客）
- 密码规则：最少 4 个字符，不作复杂度要求（本地应用，不过度设计）
- 用户名规则：2-20 个字符，中文/英文/数字/下划线
- Cookie 签名：HMAC-SHA256，secret 存储于 `data/.session_secret`

#### `mybuddy/auth/middleware.py`
**请求中间件：**

- 在每个请求中自动解析 cookie，注入 `request.state.user_id`（或 None）
- 所有 API 端点通过 `request.state.user_id` 判断用户身份
- user_id 为 None 即为访客模式

## 4. 修改现有文件（14 个文件）

### 4.1 Agent 核心（2 个文件）

#### `mybuddy/agent/core.py`——**改动最大的文件**
**现状**：ReAct 循环流程 —— 情绪检测 → 构建支持 → 匹配技能 → 检索记忆 → LLM 循环

**需要的改动**：
1. **插入安全审核门**（在情绪检测之前）：
   - 用户输入 → InputModerator → blocked 直接返回安全消息 → 不调用 LLM
2. **插入危机检测门**（在输入审核通过之后）：
   - CRITICAL/HIGH → 直接返回 CrisisResponse → 不进入 Agent 循环
3. **插入输出审核门**（在 LLM 生成完最终文本之后）：
   - 最终文本 → OutputModerator → flagged → 重写 → 返回
4. **CBT 机会检测**：每轮对话中检测是否存在自然引入 CBT 技巧的机会窗口。如有，在 system prompt 中注入对应的引导提示。CBT 技巧完全融入对话，不切换模式
5. **情绪结果持久化**：每次 `_detect_emotion()` 后调用 `MoodTracker.record()`
6. **会话关联**：将消息关联到当前活跃 Session

#### `mybuddy/agent/context.py`——**系统提示词全面改造**
**现状**：角色扮演向 system prompt（"你不是客服也不是心理咨询师"），边界设定为"不替代专业心理咨询"但同时又要求"避免'建议你寻求专业帮助'这类扫兴开场"

## 6. 数据流变化

### 6.1 当前单轮对话流程
```
用户输入 → 情绪检测 → 情感支持 → 技能匹配 → 记忆检索 → 搜索预取 → System Prompt → LLM ReAct循环 → 返回
```

### 6.2 改造后单轮对话流程
```
用户输入
  → [认证检查] ← NEW（解析 cookie，确定 user_id 或访客）
  → [输入审核] ← NEW
    ├─ BLOCKED → 安全响应 → 记录安全事件 → 返回
    └─ SAFE/FLAGGED → 继续
  → [危机检测] ← NEW
    ├─ CRITICAL/HIGH → 危机响应 → 返回
    └─ MEDIUM/LOW/NONE → 继续
  → 情绪检测（增强版：valence + category）
  → [持久化情绪记录] ← NEW（仅登录用户，访客跳过）
  → [无感化评估检查] ← NEW：检查用户回答是否覆盖某评估维度，自动评分
  → [评估维度投放判断] ← NEW：判断是否适合在此时自然引入一个未覆盖维度
  → [CBT 机会检测] ← NEW：检测是否出现引入 CBT 技巧的自然机会窗口
  → 情感支持（增强版：category 感知）
  → 技能匹配 + 记忆检索 + 搜索预取
  → System Prompt（改造版：心理健康角色 + 安全边界 + 待投放评估维度提示 + CBT 技巧引导提示）
  → LLM ReAct循环
  → [输出审核] ← NEW
    ├─ SAFE → 返回
    └─ FLAGGED → 重写 → 返回
```

### 6.3 无感化评估的独立流程
```
[后台持续运行，用户无感知]

评估周期开始（每 2 周自动重置或用户手动重置）
  ↓
每轮对话后：AssessmentScorer 检查本轮对话
  ├─ 用户回答匹配到某待定维度？→ 自动评分（0-3），维度状态 → SCORED
  └─ 没有匹配 → 跳过
  ↓
本轮对话中：如果有自然空档 + 存在 UNASKED 维度
  → System Prompt 中注入 1 个维度的自然提问指引（如"用户提到了工作，可以自然关心一下精力状态"）
  → AI 在对话中自然引出话题（用户不知道这是评估）
  → 该维度状态 → ASKED
  ↓
所有维度 SCORED？→ 评估周期完成
  ├─ 计算总分
  ├─ 如果用户开启了"状态提醒"→ 下次对话中温和提及
  └─ "我的状态"页面可查看完整结果
```

## 7. 已确认的决策

| # | 问题 | 决策 |
|---|------|------|
| 1 | 访客转登录时数据如何处理 | **用户自己选择**：登录时弹窗询问"是否导入当前对话？" |
| 2 | CBT 状态存储策略 | **登录存后端（DB），访客存前端（内存）**：登录用户 CBT 会话持久化，访客刷新即丢失 |
| 3 | 评估维度过期时间 | **2 周过期**：维度回答超过 14 天自动重置为 UNASKED |
| 4 | 旧单用户数据迁移 | **不迁移**：旧数据留在原位不动，新用户系统从零开始 |
| 5 | 移动端底部导航 | **聊天 + 我的**：两个 Tab，其他功能在聊天中自然触达或从"我的"进入 |
| 6 | AI 角色名称 | **保持 MyBuddy / 小布**：品牌延续，角色定位从"生活伙伴"调整为"懂心理学的温暖朋友" |

## 8. 仍待确认的问题

1. **数据导出**：情绪数据和评估数据是否允许导出？建议在设置页提供 JSON 导出。

2. **危机热线区域化**：是否需要根据用户语言/地区自动匹配热线号码？当前默认使用中国热线。

3. **密码找回**：本地部署场景，用户忘记密码时如何处理？建议提供管理员 CLI 重置命令。

4. **评估周期重置**：固定每 2 周自动重置，用户可手动提前触发新周期（已确定）。
