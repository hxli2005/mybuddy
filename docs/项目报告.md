# MyBuddy 项目报告

**项目名称**:MyBuddy — 生活陪伴型 AI 小伙伴
**作者**:李昊翔  **日期**:2026-06  **形态**:单用户本地可演示版本

---

## 摘要

MyBuddy 是一个运行在本地的生活陪伴型 AI Agent。与"有问有答、跨天即失忆"的单轮工具型助手不同,它的目标是**记得用户、主动关心用户、并在使用中持续变得更懂用户**。项目以工程化方式复现并落地了 NousResearch Hermes Agent 的自学习机制,核心的 ReAct 主循环与分层记忆系统均为自研。

系统具备多轮对话与角色人格编排、分层长期记忆与用户画像、8 个内置工具、情绪识别与主动关怀、定时调度与夜间记忆整理(Dream Job)、自生长的技能子系统、反馈回流闭环,以及命令行、本地 Web、QQ 官方机器人三个共享内核的接入渠道。

截至本报告,项目已完成七个里程碑、形成可每日自用的闭环。代码约 **1.2 万行**,配套测试约 **6 千行**(测试与源码比接近 2:1),**236 个测试用例全部通过**,且全部以脚本化的假 LLM 离线运行。

---

## 一、背景与动机

**产品层面。** 当前主流 AI 助手多为"单轮工具型":有问有答,但缺乏人格连贯与长期记忆,跨天对话近乎"失忆"。而"情感陪伴 + 日程提醒"是真实、高频且高度个人化的需求场景——它要求系统不仅能回答,还要**记得上下文、有稳定的人格、并在合适的时机主动出现**。本项目以"给自己每天使用"为出发点,构建"使用—反馈—改进"的闭环。

**技术层面。** 对 Hermes、Letta、mem0 等开源项目调研后的判断是:其设计思路值得借鉴,但直接 fork 的维护成本高于收益(上游 bugfix 难以及时同步、抽象与自身需求错配)。因此本项目采取**"基础设施直接复用、核心循环自研"**的策略——调度(APScheduler)、存储(SQLite / SQLAlchemy)、嵌入(API embedding)等直接采用成熟组件;而 Agent 主循环、分层记忆、技能系统自研,以保证逐层可控、可测试、可演化。

项目的差异化不在提示词,而在三件事:**记忆结构、主动性、反馈回流**。这三者共同决定了系统是"一个陪伴者"而非"又一个聊天框",也是后文技术亮点的主线。

## 二、系统架构

系统采用分层设计,各层职责清晰、目录独立:

| 层 | 模块 | 职责 |
|---|---|---|
| 接入层 | `cli` / `web` / `channels` | 命令行、本地 Web、QQ 官方机器人,三入口共享同一套内核 |
| 服务层 | `services` | `ChatService` 共享内核,统一承载多渠道对话、用户与配额 |
| 编排层 | `agent` | ReAct 主循环:构建上下文 → 思考 → 调用工具 → 观察 → 收敛 |
| 能力层 | `llm` / `tools` / `memory` / `learning` / `emotion` | Provider 抽象、工具系统、分层记忆、自学习、情绪 |
| 支撑层 | `scheduler` / `storage` | 定时调度(APScheduler)、持久化(SQLAlchemy)、主动消息队列 |
| 数据层 | SQLite / 文本档案 / JSONL | 结构化数据、`raw/conversations/archive` 文本档案、对话轨迹 |

**一次对话的处理流程**(单轮内):

1. 进入主循环前,先排空主动消息队列(可能有离线期间触发的提醒 / 问候 / nudge);
2. 用 `small_model` 对用户消息做一次**情绪分类**(label + strength);
3. 构建系统提示:注入**人格 / 关系状态 / 用户画像 / 命中的长期记忆 / 情感支持提示 / 命中的技能**;
4. **ReAct 循环**:模型思考并按需调用工具,工具结果回灌,直至收敛或达到最大步数(默认 6 步);
5. 落盘:对话写入消息表与文本档案,完整轨迹写入 JSONL(含情绪、触发技能、关联记忆);
6. 每隔 N 轮触发一次**事实抽取**;若工具被多次成功调用,异步触发**技能复盘**。

一个关键设计是**渠道与内核解耦**:各接入渠道仅负责消息收发、账号映射与事件去重,不直接操作 Agent 与记忆。因此新增渠道无需改动核心逻辑。

## 三、核心功能详解

### 3.1 多轮对话与 ReAct 主循环

对话由自研的 ReAct(Reason + Act)主循环驱动:模型在"思考—调用工具—观察结果—再思考"之间迭代,直到产出最终答复或达到步数上限。每一轮都会构建一份结构化的系统提示,把人格、关系、记忆、情绪、技能等上下文拼装进去,使回复既贴合人设又能利用历史信息。短期对话保存在滚动窗口中,保证多轮连贯。

### 3.2 角色人格与关系编排

人格不是一句"你是一个助手",而是一套结构化配置(`config.yaml` 的 `persona`),在每轮对话中由 `agent/context.py` 注入系统提示,包含:

- **基础人设**:名字、语气、语言、对用户的称呼、回应习惯;
- **角色生活状态**(`character_life`):当下在做什么、心情、最近发生的小事——让角色"像个有自己生活的人";
- **关系模型**(`relationship_model`):关系阶段 + 五个量化轴(信任 / 自在 / 俏皮 / 依赖 / 边界清晰度),以及共同的相处仪式(`shared_rituals`)和边界说明;
- **微反应与示例对话**:用具体动作和短句表达关心,避免"作为 AI""听起来你感到"这类客服腔。

关系的五个轴会被概括成一句"关系状态"注入提示,使语气随关系亲疏自然变化。人格支持**全局默认 + 单用户覆盖**(`user_personas` 表存每个用户的完整人格 JSON),让不同渠道、不同用户解析到各自的人格。

### 3.3 分层记忆与用户画像

记忆系统按数据性质分层,而非"一个向量库装一切"。整体由**短期窗口、三层文本档案、用户画像键值表、可选语义索引**四部分组成。

**(1) 短期记忆。** 最近若干轮对话的滚动窗口(默认 20 条),保证多轮连贯,随会话存续。

**(2) 长期记忆:三层文本档案。** 持久化在 `data/chroma/` 下,分三层落盘,各司其职:

| 层 | 路径 | 形式 | 用途 |
|---|---|---|---|
| raw | `raw/<日期>.jsonl` | append-only 事件流 | 保留可追溯的原始证据 |
| conversations | `conversations/<日期>.jsonl` | 按天的整轮对话 | 供回放 / 摘要 / 抽取 |
| archive | `archive/<id>.md` | Markdown + YAML frontmatter,一卡一文件 | 供检索并注入上下文 |

archive 是检索的主体。每张档案卡是一个 `.md` 文件——frontmatter 存结构化元数据,正文存记忆内容本身:

| 字段 | 含义 |
|---|---|
| `id` / `type` | 卡片 ID / 记忆类型(见下) |
| `status` | active / archived(归档后不参与召回) |
| `tags` / `keywords` | 标签与关键词,参与召回打分 |
| `importance` / `confidence` | 重要度 / 置信度(0–1),影响排序 |
| `memory_key` | 去重锚点(见下) |
| `observed_at` / `last_seen_at` / `occurrence_count` | 首次 / 最近观察到的时间、累计出现次数 |
| `created_at` / `updated_at` | 创建 / 更新时间 |
| `source` / `session_id` | 来源(extraction / user_note / …)、所属会话 |

因为是纯文本 `.md`,可以直接打开、人工核对甚至手改——这是"可解释记忆"的基础。

**(3) 记忆类型(mem_type)。** 不同性质的记忆用不同 type 区分,典型有:`profile`(关于用户的稳定事实)、`preference`(偏好)、`entity`(身边重要的人 / 宠物)、`open_thread`(未完成话题)、`shared_moment`(共同经历 / 约定)、`memory`(一般记忆片段)、`note`(用户显式记下的笔记)。`recall_memory` 检索其中 `open_thread / shared_moment / preference / profile / memory` 五类;`search_notes` 只检索 `note`。

**(4) 去重与合并(memory_key)。** 每张卡有一个 `memory_key`,由标题、触发词、内容等归一化(小写、按中文连续串 / 字母数字切分)而成,作为同一条记忆的稳定锚点。写入时若 `memory_key` 已存在,治理层走**合并**而非新建——`occurrence_count++`、刷新 `last_seen_at`,避免同一事实堆成一堆重复卡。对实体(人 / 宠物),`memory_key` 锚定"关系 + 名字":同名实体多次提到自动合并,不同关系的同名则分开存。

**(5) 用户画像。** 姓名、生日、过敏、偏好等**硬事实**以键值对存入 SQLite 的 `profile_fields`,作为权威来源,在每轮对话注入上下文;部分重要事实同时落一张 `profile` 档案卡以供召回。

**(6) 事实自抽取。** 每 N 轮对话(默认 3 轮)把最近消息交给 LLM,自动判断哪些信息"值得记住",抽取为档案卡与画像字段(借鉴 mem0)。抽取可能产生幻觉,作为已知风险——文本档案可人工审查是其缓解手段。

**(7) 召回机制:词法 + 语义混合。** 默认走**词法打分**,按查询词与卡片(正文分词 + keywords + tags)的重叠度加权,大致为:

```
score ≈ 0.45·词重叠 + 0.15·标签命中 + 0.15·关键词命中
        + 0.1·重要度 + 0.1·置信度   (整串子串命中再 +0.2,上限 1.0)
```

挂载**语义召回**后,用 API embedding 把卡片嵌入一个旁路向量索引(`vectors.db`,表 `card_vectors(card_id, content_hash, mem_type, vec)`,向量按归一化 float32 存、余弦即点积;按 `content_hash` 只重嵌变化过的卡;任何 embedding 失败都**静默降级**为纯词法,绝不阻断主流程)。再用 **RRF(Reciprocal Rank Fusion)** 融合词法与向量两路排名:`score(c) = Σ 1 / (k + rank)`(默认 `k=60`),只看排名、规避两路量纲不同的归一难题。整体策略是"**召回宽、展示窄**":每类按候选倍数(默认 4×)宽召回,合并去重重排后再截到展示条数。

### 3.4 工具系统(8 个内置工具)

工具通过装饰器注册到统一的 `ToolRegistry`,运行时上下文(配置、引擎、调度器等)按线程 / 任务隔离注入。当前内置 8 个工具:

| 工具 | 功能 | 说明 |
|---|---|---|
| `set_reminder` | 设置提醒 | 支持 ISO、`YYYY-MM-DD HH:MM`,以及"明天下午三点"等中文相对时间;写入 reminders 表并注册定时任务 |
| `recall_memory` | 检索长期记忆 | 跨五类记忆做词法 / 语义混合召回 |
| `write_note` | 记笔记 | 保存为带标题和标签的笔记,同时写入长期记忆档案 |
| `search_notes` | 搜笔记 | 在用户笔记(`mem_type=note`)里做语义搜索 |
| `weather` | 查天气 | 调用 open-meteo;失败有降级,支持 mock |
| `translate` | 翻译 | 基于 LLM 的多语种翻译 |
| `web_search` | 网页搜索 | 优先用 ddgs(浏览器指纹 HTTP 绕过 DuckDuckGo 反爬),失败回退 HTML 端点 |
| `list_skills` | 查看技能 | 列出当前在用的自学习技能 |

所有外部调用均有超时与降级,失败不阻塞主对话。

### 3.5 情绪识别与情感支持

每轮对话开头用 `small_model` 做一次轻量情绪分类,严格输出 JSON `{label, strength, reason}`:

- **label** 三选一:positive / neutral / negative;**strength** 0–1(中性 0–0.2、轻度 0.3–0.5、强烈 0.6–1.0);
- 解析失败时回退 neutral,不阻塞主对话;
- 据此生成**情感支持策略**与场景模式:强烈负向(≥0.6)进入更克制的"安全"模式、优先给安全建议;
- **情绪追踪器**记录最近窗口,**连续负向**时向系统提示追加"先放轻靠近、不要急着安排任务",并触发一条延时的关怀 nudge 进入主动消息队列。

### 3.6 主动关怀:定时调度与 Dream Job

主动性由 APScheduler 驱动,任务持久化到数据库,进程重启后自动恢复;静默时段(默认 23:00–08:00)内不推送打扰。四类定时任务:

- **每日问候**(默认 09:17):每天一条早安;
- **到期提醒**:用户设的提醒到点触发,写入主动消息队列;
- **会话沉默跟进**:用户久未回复且有合适由头时,低频补一句惦记(带冷却、每日上限、静默时段约束,避免骚扰);
- **Dream Job**(默认凌晨 02:23,也可 `mybuddy dream run` 手动触发):夜间记忆整理,做三件事——
  1. **去重合并**:对 `memory` 档案两两算文本相似度(≥0.82 视为重复),保留更精简的一条;
  2. **关怀 nudge 生成**:从未完成话题(`open_thread`)里挑有具体由头的,交给 LLM 写 1–2 条事件式短信(同一话题 3 天内不重复);
  3. **角色动态生成**:基于角色生活状态与近期关系记忆,低频生成一条不强迫回复的动态。
  每一步都在 `try/except` 中隔离,失败不影响其他步骤;执行结果汇总成 `DreamReport`(合并数 / nudge 数 / 动态数)。

### 3.7 自学习技能子系统

技能(skill)是系统从使用中**自己总结**出来的应对习惯,以人类可读、可手工编辑的 Markdown 文件存储(`data/skills/<name>.md`,frontmatter 存元数据、正文存步骤)。机制:

- 当一轮对话里**工具被成功调用三次以上并收敛**,异步让 LLM 复盘是否将该做法抽象为技能(初始置信度 0.3);
- 技能命中后注入系统提示;使用后依据反馈更新成功 / 失败计数,**置信度用 Laplace 平滑** `success / (success + fail + 1)`;
- **连续失败、置信度跌破阈值**时自动归档下线,但保留文件、可人工恢复(`mybuddy skills unarchive`)。

整个过程无需用户介入——系统在后台持续"积累经验",做对加分、做错扣分、太糟自己收起。

### 3.8 反馈回流闭环

用户的一次反馈(`/good` `/bad` `/fix`,或对话中隐含的纠错关键词)会封装成一个反馈事件,经反馈总线**同时广播到三个订阅者**:

- **轨迹**:为对应对话轮写 label 文件,为未来微调(SFT / DPO)留存数据;
- **画像 / 记忆**:调整相关条目的置信度;
- **技能**:更新对应技能的成功 / 失败计数。

负向反馈权重大于正向——"明显错误"比"回答正确"更具更新价值。三条链路解耦、订阅者之间失败隔离,每条都能独立验证到状态变化。

### 3.9 多渠道接入与共享内核

命令行、本地 Web、QQ 官方机器人三个入口共享同一套 `ChatService` 内核。以 QQ 渠道为例,它只做 **channel adapter**:负责入站事件去重、外部账号到内部用户的映射、按用户 / 按天的消息配额(默认每日 30 条),以及沙箱开关;**不直接操作 Agent / Memory / Profile**,所有对话统一进入服务层。这种解耦使得新增渠道几乎不触碰核心逻辑。

本地 Web 提供一个轻量前端,把"看不见的内部状态"可视化为五个面板:对话、画像与长期记忆、提醒与待播消息、笔记、自学习技能,以及人格设置,均可在界面上查看与编辑。

## 四、关键设计与技术亮点

前述功能背后,有四处设计是项目质量的关键支点。

### 4.1 记忆按性质分流,而非单一向量库

结构化事实(姓名、生日、过敏)以键值对存储;推测性 / 情境性的内容才进文本档案并参与召回。理由是:两类数据的更新方式、可信度、可审查性完全不同,混入同一向量库的系统难以长期维护。文本档案可人工打开核对,是"可解释记忆"的基础。

### 4.2 自学习落地为"可审查的文本资产"

技能不是黑箱权重,而是一份份带计数与置信度的 Markdown。这让"系统学到了什么"完全透明、可读、可改、可恢复,也呼应了 Hermes 的自学习理念,但以更可控的形态落地。

### 4.3 反馈真正回流到三条线

多数助手对"好 / 不好"只能道歉。本项目让每条反馈同时作用于画像、技能、轨迹,且负向权重更大。轨迹中的 label 是面向未来模型微调的标准数据接口。

### 4.4 Provider 抽象与传输健壮性

系统对 LLM 的调用收敛为唯一抽象点 `BaseLLMProvider.generate(...)`,system prompt 以独立参数传递而非混入消息序列(对本地小模型更友好,也与 Hermes 原生协议一致)。云端实现(Claude / OpenRouter)在 Provider 内部完成格式转换,对外接口统一;未来切换为本地模型(vLLM / llama.cpp)仅需新增一个 Provider 实现,业务代码无需改动——这是未来降低推理成本的关键锚点。

工程上,Provider 通过 `asyncio.to_thread` 以同步客户端发起请求,规避了部分环境下 httpx 异步传输在 TLS 阶段被重置的问题,同时不阻塞事件循环。

**规模与指标**

- 源码约 **12,300 行**,测试约 **6,000 行**,测试与源码比接近 **2:1**;
- 能力面:**8 个内置工具、10 个命令行命令、3 个共享内核的接入渠道**;
- 模块规模(行):记忆 ~2.6k、工具 ~1.3k、学习 ~1.1k、存储/智能体各 ~1k、服务 ~0.7k、调度/LLM/情绪各 ~0.4–0.5k。

**命令行命令(10 个)**:`version` `init` `chat` `web` `qqbot` `dream` `users` `profile` `reminders` `skills`。

## 六、工程质量

- **可测**:外部依赖全部可 mock——脚本化 LLM、确定性假嵌入(绕开约 2GB 的 BGE-M3 下载)、伪造的天气与搜索响应,使测试稳定、可离线运行;
- **容错**:外部调用均有降级路径,LLM 失败不阻塞主对话,Dream Job 逐步隔离,画像以 SQLite 为准、档案作可检索副本;
- **可观测**:每轮对话的完整轨迹按天落盘为 JSONL,记录情绪、触发的技能与关联记忆;
- **配置外置**:YAML 配置支持 `${ENV_VAR}` 展开,mock 开关与密钥分离;
- **分层清晰**:接入 / 服务 / 编排 / 能力 / 支撑各层独立成目录,职责单一。

正因为这样,才敢让系统在凌晨自行运行后台任务、并在第二天通过轨迹复盘问题。

## 七、功能演示

演示运行于本地 Web 前端,数据由 `scripts/seed_demo.py` 预置(围绕用户"李昊翔"与陪伴角色"小布"的一套连贯数据)。启动后浏览器访问本地地址,各界面分别对应系统的核心能力:

| 界面 | 演示内容 | 对应能力 |
|---|---|---|
| 对话 | 历史多轮对话;角色语气自然、先接住情绪再给小步建议;可现场提问触发记忆召回 | 人格连贯 + 记忆 |
| TA 记得你 | 10 项画像字段(姓名 / 偏好 / 宠物 / 过敏等) + 长期记忆档案 | 分层记忆 |
| TA 学会的 | 3 个在用技能(置信度 86% / 67% / 67%)+ 1 个已归档技能(13%,可恢复) | 自学习闭环 |
| 提醒 | 4 条定时提醒 + 2 条系统主动排入的待播消息(问候 / 关怀) | 主动关怀 |

说明:浏览上述界面**不需要联网**;仅在对话框实时发送消息时才需调用云端 LLM。演示数据可随时通过 `scripts/seed_demo.py` 重置。一个可现场展示的例子:在对话框问"你还记得我养的猫叫什么吗",系统会从长期记忆召回并答出"团子"及相关细节。

## 八、局限与展望

**当前范围**:系统目前为单用户本地流程;多用户、并发与权限隔离尚未完整设计(渠道层的账号映射与事件去重已是多用户雏形)。事实自抽取存在幻觉风险,以文本档案可审查作为缓解。技能文件的并发写入在多进程场景下需进一步处理。

**后续方向**(架构接口均已预留,属"接入"而非"重构"):

1. **HTTP API**:提供 `POST /chat`、`GET /memory/search` 等接口,让更多入口接入,放大真实使用;
2. **轨迹导出与微调**:将反馈轨迹导出为微调数据集,把反馈链的价值转化为模型层面的改进;
3. **本地模型替换**:以自有数据微调的本地模型替换云端 LLM,将单轮推理成本降至接近零。

推荐顺序为 API → 微调数据 → 本地模型。决定系统演进速度的关键,是真实使用量的增长。

## 附录:运行与复现

```bash
# 安装依赖
uv sync

# 配置 LLM 密钥
cp config.example.yaml config.yaml      # 编辑填入 api_key

# 预置 / 重置演示数据
uv run python scripts/seed_demo.py

# 启动本地 Web 演示(默认 http://127.0.0.1:8000)
uv run mybuddy web

# 命令行多轮对话
uv run mybuddy chat

# 手动触发 Dream Job(夜间记忆整理)
uv run mybuddy dream run

# 运行测试(全部离线,约数秒)
uv run pytest -q
```
